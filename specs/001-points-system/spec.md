# Feature Specification: 积分系统

**Feature Branch**: `001-points-system`  
**Created**: 2026-04-28  
**Status**: Draft  

## 概述

用户通过拉新人注册获得积分，被邀请的新人在购买后也可为邀请者增加积分。用户可在积分商城用积分兑换作业/服务，后台管理员可管理积分规则和兑换记录。

## User Scenarios & Testing

### User Story 1 - 邀请拉新获得积分 (Priority: P1)

用户通过分享专属邀请链接/码，邀请新用户注册；新用户成功注册后，邀请者立即获得积分奖励；当被邀请的新用户首次完成付款订单后，邀请者再次获得额外积分。

**Why this priority**: 这是整个积分闭环的起点，直接驱动用户增长和积分来源。

**Independent Test**: 用 A 的邀请码注册 B，B 注册后 A 积分+N；B 购买后 A 积分再+M。

**Acceptance Scenarios**:
1. **Given** 用户 A 复制邀请链接，**When** 新用户 B 通过该链接注册，**Then** A 账户立即增加注册积分。
2. **Given** B 已注册且绑定了 A 的邀请关系，**When** B 完成首笔付款订单，**Then** A 账户增加购买积分。

---

### User Story 2 - 积分商城兑换作业 (Priority: P1)

已登录用户在积分商城浏览可兑换的服务/作业套餐，选择后用积分完成兑换，系统生成兑换订单，扣减对应积分。

**Why this priority**: 积分的核心消费出口，决定积分的实际价值。

**Independent Test**: 用户有足够积分时，选择商城商品下单，积分正确扣减，兑换记录生成。

**Acceptance Scenarios**:
1. **Given** 用户积分余额 ≥ 商品所需积分，**When** 用户点击兑换，**Then** 积分扣减、兑换订单生成、余额实时更新。
2. **Given** 用户积分余额 < 商品所需积分，**When** 用户点击兑换，**Then** 提示"积分不足"，禁止兑换。

---

### User Story 3 - 查看积分账户与明细 (Priority: P2)

用户在个人中心可查看当前积分余额和完整的积分变动记录（获得/消费/原因/时间）。

**Why this priority**: 透明的积分记录提升用户信任感。

**Independent Test**: 发生任意积分变动后，明细列表正确展示该条目。

**Acceptance Scenarios**:
1. **Given** 用户已登录，**When** 进入积分页，**Then** 显示当前余额及所有历史明细（时间、变动量、原因）。

---

### User Story 4 - 管理员管理积分规则与兑换 (Priority: P2)

管理员在后台可设置拉新注册积分、拉新购买积分的数值；管理商城商品（增删改价格/积分/库存）；查看、审核（通过/拒绝）兑换记录。

**Why this priority**: 管理端保障业务灵活调整和兑换履约。

**Independent Test**: 管理员修改拉新奖励值后，新的拉新行为按新值发放积分。

**Acceptance Scenarios**:
1. **Given** 管理员在后台修改"注册奖励积分"，**When** 保存，**Then** 新触发的注册事件按新值计算。
2. **Given** 有待审核兑换订单，**When** 管理员点击"通过"，**Then** 冻结积分正式扣减、订单状态变更为"已完成"，并通知用户；点击"拒绝"时，冻结积分解冻归还，并通知用户原因（详见 FR-008b）。

---

### Edge Cases

- 同一用户被多次尝试邀请时，只绑定第一次有效邀请关系。
- 邀请者自己注册（自邀请）不得获得积分。
- 兑换后如发现积分被恶意刷取，管理员可手动冻结账户积分。
- 积分不可为负数，不可转让给他人。
- 商城商品库存为 0 时不可兑换。

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 为每个用户生成唯一邀请码/链接。
- **FR-002**: 用户注册时 MUST 支持携带邀请码，建立邀请关系。
- **FR-003**: 新用户注册成功后，系统 MUST 自动为邀请者发放"注册积分"。
- **FR-004**: 新用户首笔订单支付后，系统 MUST 自动为邀请者发放"购买积分"，同时为新用户本人发放"首购积分"。
- **FR-005**: 系统 MUST 维护每个用户的积分余额（不可为负）。
- **FR-006**: 系统 MUST 记录每笔积分变动（类型、数量、关联对象、时间）。
- **FR-007**: 系统 MUST 提供积分商城，展示两类可兑换商品：①固定套餐（直接兑换服务/作业）；②折扣券（生成优惠码可在下单时抵扣金额）。
- **FR-008**: 用户 MUST 能在积分商城提交兑换申请，系统校验积分余额后冻结对应积分，状态置为"待审核"。
- **FR-008b**: 管理员审核通过后，系统 MUST 正式扣减冻结积分，订单状态变更为"已完成"；拒绝时 MUST 解冻积分并通知用户原因。
- **FR-009**: 管理员 MUST 能配置积分规则（各事件的积分奖励数值）。
- **FR-010**: 管理员 MUST 能管理商城商品（CRUD）。
- **FR-011**: 管理员 MUST 能查看并审核兑换记录（通过/拒绝）。
- **FR-013**: 积分永久有效，系统 MUST NOT 对积分设置自动过期逻辑。
- **FR-014**: 邀请关系仅支持一层，系统 MUST NOT 对间接邀请（二级及以上）发放积分奖励。

### Key Entities

- **PointRule**: 积分规则（事件类型→奖励值），由管理员配置
- **PointBalance**: 用户积分余额（与 User 1:1）
- **PointLog**: 积分明细记录（变动类型、数量、关联对象、备注）
- **Invitation**: 邀请关系（邀请者→被邀请者，状态：已注册/已首购）
- **ShopItem**: 商城商品（名称、描述、积分价格、库存、状态）
- **RedeemOrder**: 兑换订单（用户、商品、积分、状态：待审核/已完成/已拒绝）

## Success Criteria

### Measurable Outcomes

- **SC-001**: 拉新后积分在 5 秒内到账。
- **SC-002**: 积分商城商品列表加载 < 1 秒（< 50 件商品场景）。
- **SC-003**: 积分余额在任何并发兑换下不出现负数。
- **SC-004**: 管理员可在后台 1 分钟内完成规则修改并生效。

## Assumptions

- 积分仅用于站内兑换，不涉及提现或外部平台。
- 现有用户体系（User、Order 模型）已存在，积分系统在其上扩展。
- 兑换的"作业"指平台已有的服务套餐，由管理员预先录入商城。
- 初期不做积分过期功能（待后续迭代）。
- 邀请关系仅一层（直接邀请），不支持多级裂变。

## Clarifications

### Session 2026-04-28

- Q: 购买积分发放对象是谁？ → A: 双方都奖励——邀请者拉新时获积分，被邀请新人完成首购后也获得积分。
- Q: 积分商城兑换流程是自动还是审核？ → A: 冻结积分 + 管理员审核，通过后正式扣积分，拒绝则解冻。
- Q: 积分是否有有效期？ → A: 永久有效，不过期。
- Q: 邀请层级是几层？ → A: 仅一层，A邀B、B邀C时A不获C的奖励。
- Q: 积分商城商品类型？ → A: 两者都有——固定套餐（直接兑换服务）+ 折扣券（抵扣实际订单金额）。
