# Feature Specification: 商品列表与订单结算流程

**Feature Branch**: `002-product-shop-checkout`  
**Created**: 2026-04-29  
**Status**: Draft  
**Input**: 用户注册登录后点击"立即开始"进入商品列表，选购商品，使用优惠码，通过微信/支付宝收款码完成支付，订单状态流转：已创建 → 已支付 → 已完成

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 浏览并购买商品 (Priority: P1)

已登录用户点击"立即开始"后进入商品列表页，浏览由管理员配置的服务商品（含图片、标题、描述、价格），选择一项后进入购买确认页。

**Why this priority**: 这是整条购买链路的入口，没有此流程其他步骤无法进行。

**Independent Test**: 可独立测试：用户登录后跳转到商品列表，能看到管理员配置的商品卡片，点击进入详情页即为完成。

**Acceptance Scenarios**:

1. **Given** 用户已登录，**When** 点击首页"立即开始"按钮，**Then** 跳转至商品列表页，展示所有已发布商品（含图片、名称、描述、价格）。
2. **Given** 商品列表页，**When** 用户点击某商品的"购买"按钮，**Then** 进入购买确认页，显示商品详情与价格。
3. **Given** 用户未登录，**When** 点击"立即开始"，**Then** 跳转至登录页，登录成功后重定向回商品列表。

---

### User Story 2 - 使用优惠码享受折扣 (Priority: P2)

在购买确认页，用户可在输入框中手动输入优惠码字符串，系统实时验证并预览折扣后价格，然后继续结算。

**Why this priority**: 优惠码是促销手段，影响实际支付金额，需在支付前完成输入。

**Independent Test**: 用户在购买确认页输入一个有效优惠码，点击验证后订单金额更新显示折扣后价格。

**Acceptance Scenarios**:

1. **Given** 购买确认页，**When** 用户在优惠码输入框输入有效码并点击"验证"，**Then** 系统校验通过，页面实时更新为折扣后金额，原价与折扣金额均显示。
2. **Given** 用户输入已过期或已使用的优惠码，**When** 点击验证，**Then** 系统提示具体原因（"优惠码已过期"或"优惠码已使用"）并不应用折扣。
3. **Given** 用户输入不存在的优惠码，**When** 点击验证，**Then** 系统提示"优惠码无效"。
4. **Given** 优惠码验证通过，**When** 用户继续点击"去支付"，**Then** 系统在创建订单时后端再次校验，防止并发滥用。

---

### User Story 3 - 扫码支付并通知管理员 (Priority: P1)

用户确认购买后，弹出支付弹窗，显示微信收款二维码和支付宝收款二维码，并标注"备注订单号"提示。支付完成后用户点击"我已支付"，订单状态变为"已支付"，等待管理员核对。

**Why this priority**: 支付环节是核心交付动作，直接关联订单状态变更。

**Independent Test**: 用户点击"去支付"弹出收款码弹窗，点击"我已支付"后订单状态从"已创建"变为"已支付"。

**Acceptance Scenarios**:

1. **Given** 购买确认页，**When** 用户点击"去支付"，**Then** 弹窗显示微信和支付宝收款二维码，及系统生成的唯一订单号与备注提示。
2. **Given** 支付弹窗，**When** 用户点击"我已支付"，**Then** 订单状态更新为"已支付"，弹窗关闭，页面跳转至订单详情并提示"请联系管理员核对订单"。
3. **Given** 订单已标记"已支付"，**When** 管理员在后台核对完成，**Then** 管理员可将订单状态更新为"已完成"。

---

### User Story 4 - 管理员维护商品列表 (Priority: P2)

管理员在后台可以新增、编辑、删除服务商品，设置名称、图片、描述、价格及上下架状态。

**Why this priority**: 商品内容由管理员驱动，前台列表依赖后台配置数据。

**Independent Test**: 管理员在后台新增一个商品并发布，前台商品列表立即出现该商品。

**Acceptance Scenarios**:

1. **Given** 管理员后台商品管理页，**When** 新增商品并填写完整信息后保存，**Then** 该商品出现在前台商品列表。
2. **Given** 已存在商品，**When** 管理员编辑商品信息（修改图片/价格/描述），**Then** 前台立即展示最新内容。
3. **Given** 已存在商品，**When** 管理员将其设为"下架"，**Then** 前台商品列表不再展示该商品。
4. **Given** 管理员删除某商品，**When** 该商品存在关联订单，**Then** 系统提示并阻止删除（或仅下架）。

---

### User Story 5 - 管理员制作优惠码 (Priority: P3)

管理员可在后台生成优惠码，设置折扣金额/折扣率、有效期、使用次数限制（单次单个）。优惠码生成后管理员通过其他渠道（如微信）告知用户，用户在购买页手动输入使用。

**Why this priority**: 优惠码是促销工具，可后期追加，不阻断核心支付流程。

**Independent Test**: 管理员生成一个7天有效、8折的优惠码，分配给某用户，该用户在购买页能看到此优惠码。

**Acceptance Scenarios**:

1. **Given** 管理员后台优惠码管理页，**When** 创建一个优惠码（设置折扣、有效期），**Then** 系统生成唯一码并保存。
2. **Given** 优惠码已创建，**When** 管理员通过站外渠道（如微信）将码告知用户，**Then** 用户在购买确认页的文本输入框中手动输入该码，系统验证通过后显示折扣价格。
3. **Given** 优惠码到达有效期，**When** 用户查看可用优惠码，**Then** 已过期码不显示。
4. **Given** 优惠码已被使用一次（单次限制），**When** 同一用户再次尝试使用，**Then** 系统拒绝并提示已使用。

---

### Edge Cases

- 用户在支付弹窗打开期间网络断开，点击"我已支付"后应如何处理？ → 本地乐观更新，后台异步确认，失败时提示用户重试。
- 同一优惠码被并发使用怎么办？ → 后端加唯一使用锁，先到先得，第二个用户提交时返回已使用错误。
- 管理员删除已被某用户选中（尚未结算）的商品？ → 用户侧显示"商品已下架，无法完成订单"。
- 订单创建后用户长时间不支付？ → 订单无自动过期（手动管理），管理员可手动关闭订单。

## Requirements *(mandatory)*

### Functional Requirements

**前台（用户端）**

- **FR-001**: 系统 MUST 在用户登录后展示商品列表页（含图片、名称、描述、价格）。
- **FR-002**: 系统 MUST 允许用户在购买确认页手动输入优惠码字符串，验证通过后实时预览折后价；输入无效/过期/已用码时显示具体错误提示。
- **FR-003**: 系统 MUST 在用户点击"去支付"时弹出含微信和支付宝收款二维码的弹窗，并展示订单号与备注提示。
- **FR-004**: 系统 MUST 在用户点击"我已支付"后将订单状态更新为"已支付"，并向管理员发送站内通知（含订单号、商品名、实付金额）；用户侧跳转至订单详情页并展示"等待管理员核对"提示。
- **FR-005**: 系统 MUST 展示用户的订单列表，状态包括：已创建、已支付、已完成、已取消。
- **FR-006**: 系统 MUST 在优惠码被选用后将其标记为已使用，防止重复使用。

**后台（管理员端）**

- **FR-007**: 管理员 MUST 能够新增、编辑、删除、上下架商品（含图片上传、名称、描述、价格）。
- **FR-008**: 管理员 MUST 能够创建促销优惠码（PromoCoupon），设置折扣值（固定金额或百分比）、有效期、使用次数（单次）；促销优惠码为通用型，不绑定特定用户，任何持有该码的用户均可使用一次。
- **FR-009**: 管理员 MUST 能够查看所有订单，筛选状态，将"已支付"订单更新为"已完成"，并可将任意未完成订单标记为"已取消"。
- **FR-011**: 系统 MUST 向管理员后台推送站内通知，当订单状态变为"已支付"时触发，通知内容包含订单号、商品名、实付金额；管理员后台顶栏显示铃铛图标与未读角标，点击展开最近通知下拉列表，支持标记已读。
- **FR-010**: 管理员 MUST 能够查看收款二维码配置并更新微信/支付宝收款码图片。

### Key Entities

- **商品（Product）**: 名称、图片、描述、价格、状态（上架/下架）、创建时间。
- **订单（Order）**: 关联用户、关联商品、使用的优惠码（可空）、原价、实付价格、状态（已创建/已支付/已完成/已取消）、系统生成的唯一订单号、创建时间、取消原因（可空）。
- **促销优惠码（PromoCoupon）**: 唯一码字符串、折扣类型（固定金额/百分比）、折扣值、有效期开始/结束、使用次数上限（默认1次）、已使用次数（通用码，不绑定用户）。注：与积分系统（Feature 001）中由兑换生成的 `Coupon` 是两套独立模型，前者由管理员直接创建，后者由用户积分兑换产生。
- **收款码配置（PaymentConfig）**: 微信收款码图片URL、支付宝收款码图片URL、最后更新时间。
- **站内通知（Notification）**: 接收方（管理员）、通知类型（新支付订单）、内容摘要、关联订单ID、是否已读、创建时间。

---

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从点击"立即开始"到看到商品列表的时间不超过 2 秒。
- **SC-002**: 用户完成"选商品 → 确认 → 弹窗支付 → 标记已支付"全流程不超过 3 分钟。
- **SC-003**: 优惠码的有效性校验在提交时 100% 准确（不允许使用已过期或已用码）。
- **SC-004**: 管理员在后台更新商品信息后，前台在 5 秒内反映最新数据。
- **SC-005**: 订单状态变更后，用户订单列表页面刷新后立即显示最新状态。

## Clarifications

### Session 2026-04-29

- Q: 新商品/订单系统与现有订单系统是取代、并存还是融合关系？ → A: 取代。新商品购买流程完全替换现有"提交需求"订单流程，删除旧的需求类型（OrderType）和旧版订单逻辑，仅保留一套订单系统。
- Q: 订单是否需要"已取消"状态？ → A: 需要。状态机为 已创建 → 已支付 → 已完成，管理员可将任意未完成订单标记为"已取消"作为终止状态。
- Q: 优惠码获取方式是下拉框分配还是用户手动输入？ → A: 仅手动输入。购买确认页去掉下拉框，改为文本输入框，用户手动输入优惠码字符串，后端校验有效性（未过期、未使用、适用该商品）。
- Q: 支付后如何引导用户联系管理员？ → A: 站内消息。用户点击"我已支付"后，系统自动向管理员发送一条站内通知（含订单号、商品名、实付金额），同时用户侧跳转至订单详情页提示等待管理员核对。
- Q: 管理员后台通知如何展示？ → A: 顶栏铃铛图标，显示未读角标，点击展开最近通知下拉列表，可标记已读，不新增侧边栏菜单。

## Assumptions

- 支付方式为"线下扫码转账"，系统不接入第三方支付 SDK，由用户手动扫码并备注订单号。
- 管理员人工核对支付后在后台手动将订单改为"已完成"，无自动对账。
- 收款二维码为静态图片，由管理员在后台上传，所有商品共用同一套收款码。
- 每笔订单一次只能购买一个商品（单品订单），不支持购物车多品结算。
- 优惠码为单次单个使用，不支持叠加。
- 图片存储使用现有的图片上传机制（与轮播图管理一致）。
- 现有用户认证系统（JWT）直接复用，无需新增认证逻辑。
- 新商品购买订单系统**取代**现有"提交需求"流程；旧的 OrderType（需求类型）、旧版 Order 及相关前后台页面将被删除并由新系统替代。
