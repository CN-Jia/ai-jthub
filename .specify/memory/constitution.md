# ai-jthub Constitution

## Core Principles

### I. 代码质量优先（Code Quality First）

- 所有代码必须清晰、可读、可维护，避免过度设计
- 函数/方法单一职责，每个模块只做一件事
- 禁止重复代码（DRY 原则），公共逻辑必须抽取复用
- 代码提交前须通过 lint 检查，保持风格统一
- 复杂逻辑必须添加注释说明意图，而非描述"做了什么"
- 变量、函数命名要语义清晰，禁止无意义命名（如 `a`、`temp`、`data2`）

### II. 测试覆盖（Testing Coverage）

- 关键业务逻辑（状态流转、订单校验、权限鉴别）必须有对应单元测试
- 推荐覆盖率目标：核心 service 层 ≥ 50%（个人项目，务实优先）
- 边界条件与异常场景应有测试用例，不要求 100% 分支覆盖
- 测试必须可独立运行，不依赖外部数据库状态或执行顺序
- 禁止为了让测试通过而修改测试逻辑（除非需求变更）

### III. 用户体验一致性（UX Consistency）

- 界面交互遵循一致的视觉语言和操作范式
- 所有用户可见的错误信息必须友好、清晰，不暴露技术细节
- 关键操作提供加载状态、成功/失败反馈
- 响应时间：页面首屏加载 < 2s，用户交互响应 < 300ms
- 无障碍访问（Accessibility）：关键 UI 组件支持键盘导航和屏幕阅读器
- 移动端与桌面端适配，优先移动端设计

### IV. 性能与可观测性（Performance & Observability）

- 避免不必要的渲染、重复请求和内存泄漏
- 关键路径操作需要性能基准测试
- 结构化日志：记录关键操作、错误堆栈、性能指标
- 错误必须被捕获并上报，禁止静默吞掉异常
- 依赖项保持最小化，引入新库前评估必要性和包体积影响

### V. 简洁性（Simplicity — YAGNI）

- 只实现当前需求，不为"未来可能用到"的功能提前设计
- 优先使用标准库和已有工具，减少自造轮子
- 架构决策需有明确理由，复杂性必须被证明是必要的
- 定期重构，消除技术债务，保持代码库整洁

## 安全要求（Security Requirements）

- 不在代码仓库中硬编码密钥、密码、Token 等敏感信息
- 用户输入必须进行验证和转义，防止注入攻击
- 依赖库定期检查安全漏洞（`npm audit` / `pip audit`）
- `.github/` 目录中的 Agent 配置文件不得包含凭证信息，建议加入 `.gitignore`

## 开发工作流（Development Workflow）

- 功能开发遵循 Spec-Driven Development 流程：constitution → specify → plan → tasks → implement
- 每个功能在独立分支开发，分支命名格式：`NNN-feature-name`
- 提交信息清晰描述变更目的，格式：`type: description`（如 `feat: add photo album drag-drop`）
- 合并前必须通过所有自动化测试
- 实现完成后运行 `/speckit.analyze` 做跨文档一致性检查

## Governance

- 本宪法优先于所有其他开发实践和约定
- AI Agent 在生成代码时必须遵守以上所有原则
- 宪法修订需说明变更原因，并更新版本号和日期
- 所有技术决策如与宪法冲突，以宪法为准
